# Test doubles

### What are they? how to use them? When to use them?

### Who is this article for?

This article is designed for:

- Beginners exploring the world of automated testing.
- Developers new to Test-Driven Development (TDD) who want to understand what makes a good test.

### What this article won't cover

While this article provides a strong foundation, it won't cover:

- TDD processes.
- Strategies like the Test Pyramid.
- Types of tests (e.g., unit vs. integration).

Instead, we'll focus on the characteristics that define a great automated test and how **Test Doubles** play a role in achieving that.

## Introduction

In software development, testing is more than a safety net—it's crucial for faster delivery, higher code quality, and greater system confidence. However, as systems become more complex, dependencies create challenges that can make tests slower, less reliable, and harder to manage. This is where **Test Doubles** become essential.

Test Doubles simulate real-world dependencies like databases, external APIs, and services. They help developers create fast, reliable, and focused tests. By isolating the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)), these tools ensure tests validate only the intended behavior without external interference.

Test Doubles emerged in the 1990s alongside early testing frameworks to address the need for component isolation. While tools like JUnit introduced systematic ways to verify code behavior, growing system complexity demanded better solutions. Test Doubles filled this need by allowing developers to test components independently without waiting for full integration.

Today, Test Doubles are essential in modern software development, enabling teams to:

- Isolate the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)) for precise behavior verification.
- Speed up test execution by eliminating dependency delays.
- Improve test maintainability by reducing coupling-related fragility.

This article explores Test Doubles in depth: their types, practical applications, and best practices. We'll show you how to avoid common pitfalls like **"The Mockery"** and write tests that are both robust and maintainable.

## What are test doubles?

Test Doubles are specialized objects that replace real dependencies in automated testing. These dependencies might be databases, external APIs, third-party libraries, or other application components. By using Test Doubles, developers can test in isolation and focus purely on the behavior and outputs of the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)).

The concept gained popularity during the early days of Test-Driven Development (TDD) when developers needed ways to decouple tests from unreliable or slow dependencies. What began as simple placeholders has evolved into sophisticated tools for simulating complex behaviors and interactions.

Their importance lies in their ability to:

- Reduce test fragility by eliminating external dependencies.
- Create deterministic tests with consistent outcomes.
- Accelerate test execution by avoiding time-consuming operations.
- Enable edge case testing that would be difficult or impossible with real dependencies.

In the following sections, we'll explore the different types of Test Doubles, their practical applications, and how to use them effectively.

## Types of test doubles

Test Doubles are categorized based on their purpose and functionality. Below, we'll break down each type with practical examples in **Java**.

### 1. **Dummy**

A **Dummy Test Double** is the simplest form of a test double, serving as a placeholder. It exists solely to fulfill method or constructor argument requirements without participating in the test logic. Dummies remain inactive during test execution, merely satisfying compiler or runtime requirements.

### Characteristics of dummy test doubles

- **Placeholders**: They exist only to satisfy method signatures or constructors.
- **Never Called**: Their methods are not used during the test.
- **Simplistic Implementation**: They contain minimal or no logic, serving a purely structural purpose.
- **Low Maintenance**: They rarely need updates during refactoring since they don't participate in test behavior.

### When to use dummy test doubles

- **Constructor or Dependency Injection**: When a class under test requires non-optional dependencies that are irrelevant to the test.
- **Focus on the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)**: When you need to prevent irrelevant objects from interfering with testing the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)).

### A simple example

```java
class BankAccount {
    void processTransaction(Account account, Transaction transaction) {
        // Logic to process the transaction
    }
}
```

```java
@Test
void testTransactionProcessing() {
    Account dummyAccount = new Account(); // Dummy object
    Transaction dummyTransaction = new Transaction(); // Additional dummy object
    BankAccount processor = new BankAccount();
    processor.processTransaction(dummyAccount, dummyTransaction);
}
```

### A more elaborate example

Let's consider a **Library Management System** where the `BookBorrowingService` handles book borrow requests. The service requires a `NotificationService` to notify users about their request status. In our test, we focus solely on verifying the borrowing logic and don't need the `NotificationService` to perform any real action.

**Production code**

```java
public interface NotificationService {
    void notify(String userId, String message);
}
```

```java
public class BookBorrowingService {
    private final NotificationService notificationService;

    public BookBorrowingService(NotificationService notificationService) {
        this.notificationService = notificationService;
    }

    public String borrowBook(String bookId, String userId) {
	// The business logic
	//...

        // Not the behavior we want to test, Notify user about the borrowing status.
        notificationService.notify(
	        userId,
	        "Book " + bookId + " borrowed successfully."
	      );

        return "Borrowed: " + bookId;
    }
}
```

Dummy **test double implementation**

```java
public class DummyNotificationService implements NotificationService {
    @Override
    public void notify(String userId, String message) {
        // Do nothing - Dummy implementation
    }
}
```

**Test code**

```java
class BookBorrowingServiceShould {

    @Test
    void borrow_book_successfully_when_inputs_are_valid() {
        // Arrange
        NotificationService dummyNotificationService = new DummyNotificationService();
        BookBorrowingService borrowingService = new BookBorrowingService(dummyNotificationService);

        // Act
        String result = borrowingService.borrowBook("12345", "user001");

        // Assert
        assertEquals("Borrowed: 12345", result);
    }
}
```

### The benefits of dummy test doubles

- **Reduced Complexity**: Simplifies test setup by eliminating irrelevant dependencies.
- **Focused Tests**: Emphasizes the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)'s behavior rather than its collaborators.
- **Minimal Impact During Refactoring**: Since dummies contain no logic, they rarely need updates when collaborators change.

### Common pitfalls

- **Overuse**: Use dummies only when dependencies are truly irrelevant to the test—misuse leads to incomplete or misleading tests.
- **Unclear Intent**: Always name and document dummies clearly to prevent confusion with actual collaborators.

When used thoughtfully, Dummy Test Doubles simplify your test setup and keep tests focused on the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)'s behavior without distractions.

### 2. **Stub**

A **Stub Test Double** provides predefined responses to method calls during testing. Unlike a Dummy, a Stub actively participates in test logic by returning controlled data that the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)) needs. Stubs isolate the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) by simulating its collaborators' behavior predictably.

### Characteristics of stub test doubles

- **Predefined Responses**: Stubs return specific data needed for test execution.
- **Focus on Inputs**: They control inputs to the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) without testing the collaborator itself.
- **Simplistic Logic**: Stubs contain minimal logic, just enough to return predefined values.
- **Passive Role**: Stubs provide data without verifying interactions or behavior.

### When to use stub test doubles

- **Input Control**: When the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)'s behavior depends on specific collaborator input.
- **Isolation of the SUT**: To prevent external dependencies from affecting test results.
- **Avoiding Non-Deterministic Factors**: To replace unpredictable elements like random values, network calls, or database queries.

### A simple example

```java
public class WeatherApp {
    private final WeatherService weatherService;

    public WeatherApp(WeatherService weatherService) {
        this.weatherService = weatherService;
    }

    public String getWeatherMessage() {
        boolean isSunny = weatherService.isSunny();
        return isSunny ? "It's sunny today!" : "It's not sunny today.";
    }
}
```

```java
public class IsSunnyStubWeatherService implements WeatherService {
    @Override
    public boolean isSunny() {
        return true;
    }
}
```

```java
public class NotSunnyStubWeatherService implements WeatherService {
    @Override
    public boolean isSunny() {
        return false;
    }
}
```

```java
class WeatherAppTest {

    @Test
    void shouldDisplaySunnyMessageWhenWeatherIsSunny() {
        // Arrange
        WeatherService sunnyStub = new IsSunnyStubWeatherService();
        WeatherApp weatherApp = new WeatherApp(sunnyStub);

        // Act
        String message = weatherApp.getWeatherMessage();

        // Assert
        assertEquals("It's sunny today!", message);
    }

    @Test
    void shouldDisplayNotSunnyMessageWhenWeatherIsNotSunny() {
        // Arrange
        WeatherService notSunnyStub = new NotSunnyStubWeatherService();
        WeatherApp weatherApp = new WeatherApp(notSunnyStub);

        // Act
        String message = weatherApp.getWeatherMessage();

        // Assert
        assertEquals("It's not sunny today.", message);
    }
}
```

### A more elaborate example

Consider a `UserService` that retrieves user information from a `UserRepository`. To test `UserService`, we can use a Stub Test Double to control the repository's returned data.

**Production code**

```java
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public String getUserGreeting(String userId) {
        User user = userRepository.findById(userId);
        if (user == null) {
            return "User not found";
        }
        return "Hello, " + user.getName();
    }
}
```

Stub **test double implementation**

```java
public class StubUserRepositoryWithUser implements UserRepository {
    @Override
    public User findById(String userId) {
			return new User("123", "John Doe");
    }
}
```

```java
public class StubUserRepositoryWithoutUser implements UserRepository {
    @Override
    public User findById(String userId) {
			return null;
    }
}
```

**Test code** using a stub

```java
class UserServiceTest {

    @Test
    void shouldReturnGreetingForExistingUser() {
        // Arrange
        UserRepository stubUserRepository = new StubUserRepositoryWithUser();
        UserService userService = new UserService(stubUserRepository);

        // Act
        String greeting = userService.getUserGreeting("123");

        // Assert
        assertEquals("Hello, John Doe", greeting);
    }

    @Test
    void shouldReturnUserNotFoundForNonExistingUser() {
        // Arrange
        UserRepository stubUserRepository = new StubUserRepositoryWithoutUser();
        UserService userService = new UserService(stubUserRepository);

        // Act
        String greeting = userService.getUserGreeting("123");

        // Assert
        assertEquals("User not found", greeting);
    }
}
```

### The benefits of stub test doubles

- **Controlled Test Environment**: Provides predictable and reproducible test results by isolating the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) from its dependencies.
- **Focus on Behavior**: Enables verification of the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)'s behavior using known inputs.
- **Simplified Setup**: Removes the need for complex or real implementations of external systems.

### Common pitfalls

- **Overuse**: Excessive use of stubs can create brittle tests that are tightly coupled to specific implementations.
- **Loss of Realism**: Stubs may miss important edge cases or unexpected scenarios that exist in real collaborators.
- **Complex Stubs**: Adding too much logic to stubs can introduce bugs in the test code itself.

When used effectively, Stub Test Doubles help streamline test setup and maintain focus on the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)'s behavior, leading to robust, isolated, and meaningful tests.

A Stub provides predetermined responses to specific method calls during testing, giving you control over how dependencies behave.

### What Is a Temporary Test Stub?

A **Temporary Test Stub** is a variant of Stub Test Double that exists only during a test or test suite. It provides controlled responses to method calls within the test's scope. Unlike permanent stubs or mock configurations, temporary test stubs are typically created inline and narrowly scoped to minimize long-term codebase dependencies.

These stubs excel in scenarios where the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)) needs a dependency to return predictable values but doesn't warrant a full permanent implementation.

### Characteristics of Temporary Test Stubs

- **Short-lived**: Exists only for a single test or suite.
- **Predefined behavior**: Predictably returns specific data.
- **Lightweight Implementation**: Written inline within tests, eliminating separate class files.
- **Encapsulation**: Contains test logic locally, preventing dependency leaks into production code.

### When to Use Temporary Test Stubs

- **Quick setup**: For controlling dependency behavior without creating full class implementations.
- **Reducing boilerplate**: To minimize codebase clutter from single-use stub classes.
- **Focused scope**: When dependency behavior is unique to specific tests.

### Example of a temporary test stub

Let's consider a `PaymentService` that interacts with an `ExchangeRateProvider`. If we want to test the behavior of the `PaymentService` when converting payment amounts, we can use a Temporary Test Stub for the `ExchangeRateProvider`.

### Production Code

```java
public class PaymentService {
    private final ExchangeRateProvider exchangeRateProvider;

    public PaymentService(ExchangeRateProvider exchangeRateProvider) {
        this.exchangeRateProvider = exchangeRateProvider;
    }

    public double convertPayment(double amount, String fromCurrency, String toCurrency) {
        double rate = exchangeRateProvider.getExchangeRate(fromCurrency, toCurrency);
        return amount * rate;
    }
}
```

### Temporary Test Stub Implementation (Inline)

```java
class PaymentServiceTest {

    @Test
    void shouldConvertPaymentUsingProvidedExchangeRate() {
        // Arrange: Temporary Test Stub
        ExchangeRateProvider temporaryStub = new ExchangeRateProvider() {
            @Override
            public double getExchangeRate(String fromCurrency, String toCurrency) {
	            return 0.85;
            }
        };
        PaymentService paymentService = new PaymentService(temporaryStub);

        // Act
        double convertedAmount = paymentService.convertPayment(100, "USD", "EUR");

        // Assert
        assertEquals(85.0, convertedAmount, 0.001);
    }
}
```

### Benefits of temporary test stubs

- **Fast and Lightweight**: Minimizes setup time by avoiding dedicated stub classes.
- **Localized Logic**: Contains test-specific behavior within the test for better readability.
- **Quick Iterations**: Enables rapid testing and experimentation.

### Common pitfalls

- **Limited reusability**: Test-specific nature may lead to duplication across similar tests.
- **Code clutter**: Overuse of inline stubs can reduce test clarity.
- **Overcomplication**: Complex stub behavior can become difficult to maintain.
- Impact on the readability of the test

Temporary Test Stubs offer efficient dependency control in isolated tests without adding unnecessary complexity. They shine in one-off tests where full-stub implementations would be overkill.

### 3. **Spy**

A **Spy Test Double** is a test double that both provides responses like a Stub and records how it was used during testing. Spies track which methods were called, their call frequency, and their parameters. They excel at testing indirect outputs—cases where the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)) interacts with a dependency without returning data.

### Characteristics of Spy Test doubles

- **Calls real implementation**: Spies wrap the real dependency, ensuring the actual behavior is executed while recording interactions. (For some reason most of the testing libraries do not implement this behavior, I’ve only seen it in javascript)
- **Captures Interactions**: Records method calls, parameters, and call counts for verification.
- **Used for verification**: While Stubs focuses on providing inputs, Spies verifies outputs and side effects.
- **Combination of roles**: Acts as both a Stub (responding) and a monitor for interactions.
- **Supports behavioral testing**: Ensures the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) correctly interacts with its collaborators.

### When to use spy test doubles

- **Behavior verification**: When you need to confirm a specific method calls on a dependency.
- **Testing side effects**: To verify proper interactions between the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) and its dependencies while still relying on their real behavior.
- **Augmenting eal implementations**: To sidestep complex dependencies while maintaining behavior verification.

### Example of a spy test double

**Production code**

```java
public class OrderService {
    private final NotificationService notificationService;

    public OrderService(NotificationService notificationService) {
        this.notificationService = notificationService;
    }

    public void placeOrder(String orderId) {
        // Business logic for processing the order
        // ...

        // Notify the customer
        notificationService.notify(
	        "Order " + orderId + " has been placed successfully."
	      );
    }
}
```

**Use of a spy** (using the Mockito testing library)

```java
class OrderServiceTest {

    @Test
    void shouldNotifyCustomerWhenOrderIsPlaced() {
        // Arrange
        NotificationService notificationService = spy(NotificationService.class);
        OrderService orderService = new OrderService(notificationService);

        // Act
        orderService.placeOrder("12345");

        // Assert
        verify(
	        notificationService,
	        times(1)
	      ).notify("Order 12345 has been placed successfully.");
    }
}
```

### Example of an implemented spy test double

Let's consider an `OrderService` that processes an order and notifies a `NotificationService` upon completion. We'll use a Spy to verify that the `NotificationService` was called correctly.

**Production code**

```java
public class OrderService {
    private final NotificationService notificationService;

    public OrderService(NotificationService notificationService) {
        this.notificationService = notificationService;
    }

    public void placeOrder(String orderId) {
        // Business logic for processing the order
        // ...

        // Notify the customer
        notificationService.notify("Order " + orderId + " has been placed successfully.");
    }
}
```

Spy **test double implementation**

```java
public class SpyNotificationService implements NotificationService {
    private final NotificationService realNotificationService;
    private final List&lt;String&gt; notificationsSent = new ArrayList&lt;&gt;();

    public SpyNotificationService(NotificationService realNotificationService) {
        this.realNotificationService = realNotificationService;
    }

    @Override
    public void notify(String message) {
        realNotificationService.notify(message); // Call the real implementation
        notificationsSent.add(message);         // Record the interaction
    }

    public int getInvocationCount() {
        return notificationsSent.size();
    }

    public List&lt;String&gt; getNotificationsSent() {
        return new ArrayList&lt;&gt;(notificationsSent);
    }

    public boolean wasCalledWith(String message) {
        return notificationsSent.contains(message);
    }
}
```

**Test code** using a spy

```java
class OrderServiceTest {

    @Test
    void shouldNotifyCustomerWhenOrderIsPlaced() {
        // Arrange
        NotificationService notificationService = new NotificationService();

        SpyNotificationService spyNotificationService = new SpyNotificationService(
	        notificationService
        );
        OrderService orderService = new OrderService(spyNotificationService);

        // Act
        orderService.placeOrder("12345");

        // Assert
        assertEquals(1, spyNotificationService.getInvocationCount());
        assertTrue(
            spyNotificationService.wasCalledWith("Order 12345 has been placed successfully.")
        );
    }
}
```

### Benefits of Spy Test Doubles

- **Verification of Behavior**: Confirms proper interactions between the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) and its collaborators.
- **Focused on Indirect Outputs**: Ideal for testing actions rather than returned values.
- **Detailed Interaction Records**: Provides comprehensive verification of calls and parameters.

### Common Pitfalls

- **Overuse of Interaction Testing**: Too much focus on interactions creates brittle tests tied to implementation details.
- **Confusion with Mocks**: While similar, Spies record interactions for later verification, whereas Mocks contain built-in assertions.
- **Complexity in Spies**: Custom Spies may add unnecessary complexity when existing frameworks like Mockito would suffice.

### 4. **Mock or Mock Object**

A **Mock Object** is a test double that simulates a collaborator while containing built-in assertions to verify interactions during testing. Unlike Spies that passively record interactions, Mocks actively verify that predefined interactions occur as expected. They are particularly useful in **behavior-driven testing**, where we need to ensure the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)) correctly interacts with its dependencies.

### Characteristics of Mock Objects

- **Interaction Verification**: Mocks verify method calls, their frequency, and parameters during testing.
- **Predefined Expectations**: Mock behavior is set up before test execution to ensure specific interactions occur.
- **Behavior-Driven**: Mocks focus on the *behavior* of the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) rather than its state.
- **Immediate Feedback**: Tests fail instantly when interactions don't match expectations.

### When to Use Mock Objects

- **Behavior-Driven Development (BDD)**: When testing [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) behavior through its interactions with collaborators.
- **Complex Interactions**: To verify specific sequences of method calls under certain conditions.
- **Isolation of Dependencies**: When real collaborators are unavailable or impractical for testing.

### Example of a Mock Object

Imagine a system where a `NotificationService` must send an email when a user successfully registers. However, the notification system itself does not affect the registration process, and we only care about ensuring that `NotificationService` is called with the correct arguments. This is a case where a **Mock** is the most appropriate choice.

**Production code**

```java
public class UserRegistrationService {
    private final NotificationService notificationService;
    private final Logger logger;

    public UserRegistrationService(
	    NotificationService notificationService,
	    Logger logger
	  ) {
        this.notificationService = notificationService;
        this.logger = logger;
    }

    public void registerUser(String email) {
        // Log the user registration
        logger.info("User registered with email: " + email);

        // Notify the user
        notificationService.sendNotification(email, "Welcome to the platform!");
    }
}
```

```java
public interface NotificationService {
    void nofify(String email, String message);
}
```

**Test code** using a mock object with Mockito

```java
public class DummyLogger extends Logger {
    protected DummyLogger() {
        super("DummyLogger", null);
    }

    @Override
    public void info(String message) {
        // Do nothing
    }
}
```

```java
class UserRegistrationServiceTest {

    @Test
    void shouldSendWelcomeEmailUponRegistration() {
        // Arrange
        NotificationService mockNotificationService = mock(NotificationService.class);
        Logger dummyLogger = new DummyLogger();
        UserRegistrationService registrationService = new UserRegistrationService(
	        mockNotificationService,
	        dummyLogger
	      );

        // Act
        registrationService.registerUser("user@example.com");

        // Assert
        verify(mockNotificationService, times(1))
                .sendNotification("user@example.com", "Welcome to the platform!");
    }
}
```

### Why Is this example ideal for a mock object?

- **Behavior Verification**:
	- The test verifies that the `notify` method is called exactly once with the correct parameters.
	- The focus is on verifying the interaction, not the outcome of the `NotificationService`.
- **Not a Stub**:
	- Unlike a Stub, which would provide predefined responses, we only care that the `NotificationService` was invoked correctly, not what it returns.
- **Not a Spy**:
	- While Spies record interactions for later verification and often work with real implementations, a Mock directly verifies interactions, making it simpler for this case.
- **Interaction-Focused**:
	- The test validates the collaboration between services without concern for the `NotificationService`'s internal state or output—exactly what Mocks excels at.

### Benefits of mock objects

- **Behavior Verification**: Ensures the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) interacts correctly with its dependencies.
- **Immediate Feedback**: Fails tests instantly if expectations aren't met, enabling quick validation.
- **Isolation**: Tests the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) independently from actual dependency implementations.

### Common pitfalls

- **Over-Mocking**: Excessive mocking creates brittle tests that couple too tightly to implementation details.
- **False Sense of Security**: Mocks verify interactions but not the actual logic or state of collaborators.
- **Setup Complexity**: Complex interaction expectations can make tests difficult to maintain.

### 5. **Fake**

### What Is a fake test double?

A **Fake Test Double** provides a lightweight implementation of a dependency, typically as a simplified or in-memory version of the real behavior. Unlike Stubs or Spies which focus on specific responses or interactions, Fakes are functional—they mirror actual component behavior in a way that's faster and more predictable for testing. They are usually hand-made

### Characteristics of Fake Test Doubles

- **Functional Implementation**: Fakes contain logic that mimics real-world functionality in a simplified form.
- **Stateful**: Unlike Dummies or Stubs, Fakes maintain state across calls, supporting complex testing scenarios.
- **Replacement for External Systems**: Fakes substitute for databases, APIs, or services, creating a controlled testing environment.
- **Improved Performance**: Fakes run faster and more reliably than real dependencies, especially in integration tests.

### When to Use Fake Test Doubles

- **Simulating Complex Dependencies**: When testing systems with stateful or complex dependencies.
- **Replacing External Systems**: To eliminate the overhead of databases, file systems, or third-party APIs.
- **Testing in Isolation**: When you need realistic behavior without a full production setup.
- **Integration Testing**: To test component interactions without actual infrastructure.

### Example of a Fake Test Double

Consider a `TaskManager` that tracks tasks and interacts with a `TaskRepository`. Instead of connecting to a real database, we use a Fake that simulates a repository with in-memory storage.

Production Code

```java
public class TaskManager {
    private final TaskRepository taskRepository;

    public TaskManager(TaskRepository taskRepository) {
        this.taskRepository = taskRepository;
    }

    public void addTask(Task task) {
        taskRepository.save(task);
    }

    public List&lt;Task&gt; getAllTasks() {
        return taskRepository.findAll();
    }
}
```

### Fake Test Double Implementation

```java
public class FakeTaskRepository implements TaskRepository {
    private final List&lt;Task&gt; tasks = new ArrayList&lt;&gt;();

    @Override
    public void save(Task task) {
        tasks.add(task);
    }

    @Override
    public List&lt;Task&gt; findAll() {
        return new ArrayList&lt;&gt;(tasks);
    }
}
```

### Test Code Using a Fake

```java
class TaskManagerTest {

    @Test
    void shouldAddAndRetrieveTasks() {
        // Arrange
        TaskRepository fakeTaskRepository = new FakeTaskRepository();
        TaskManager taskManager = new TaskManager(fakeTaskRepository);
        Task task = new Task("Test Task");

        // Act
        taskManager.addTask(task);
        List&lt;Task&gt; tasks = taskManager.getAllTasks();

        // Assert
        assertEquals(1, tasks.size());
        assertEquals("Test Task", tasks.get(0).getName());
    }
}
```

### Benefits of Fake Test Doubles

- **Realistic Behavior**: Fakes simulate real-world functionality, creating a test environment that mirrors production.
- **Isolation from External Dependencies**: Fakes provide a reliable alternative to real system interactions.
- **Faster Execution**: Tests using Fakes run faster than those using actual systems.
- **Stateful Testing**: Fakes support testing scenarios that need state tracking across operations.

### Common Pitfalls

- **Increased Maintenance**: Fakes need updates as the real system evolves.
- **Mismatch with Real Systems**: Inaccurate Fakes can lead to passing tests but production failures.
- **Overuse**: Using Fakes too liberally can blur the distinction between test doubles and full implementations.
- **Testing structure not behavior**.

### Additional Insights

- **Behavior and state**:
	- Only **Spies** and **Fakes** maintain state, tracking or simulating interactions over time.
	- **Dummies**, **Stubs**, and **Temporary Stubs** are stateless by design.
- **Purpose and focus**:
	- Dummies serve as structural placeholders, while Stubs, Spies, Mocks, and Fakes emulate or verify behavior.
	- Fakes offer the most realistic dependency simulation, making them ideal for integration-like tests, while others target unit testing.
- **Common pitfalls**:
	- **Confusion between types**:
		- Stubs and Mocks overlap in providing behavior and verifying interactions, leading to confusion.
		- Spies and Mocks both verify interactions but differ in their approach and strictness.
	- **Overuse**: Mocks and Spies can create brittle tests if they focus too much on implementation details.
	- **Complexity**: Fakes need constant maintenance to stay realistic and correct as systems change.
- **Choosing the Right Double**:
	- **Who is the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)?**
	- **What behavior or interaction am I testing?**
	- **Do I need to simulate behavior or verify interactions?**
	- **Do I need a realistic state or simplified responses?**

### Introduction to Command Query Separation and Test Doubles

**Command Query Separation (CQS)** is a fundamental principle in software design introduced by Bertrand Meyer. It states that every method in a system should be either a **Command** or a **Query**, but never both:

- **Commands** are operations that perform an action or change the system's state but do not return data (e.g., `placeOrder`).
- **Queries** are operations that return data without modifying the system's state (e.g., `getOrderDetails`).

This principle helps in building systems that are easier to understand, maintain, and test. By clearly separating actions that modify the state from those that retrieve information, CQS reduces cognitive load and potential side effects, leading to more predictable and reliable behavior.

## CQS in the context of test doubles

When working with **Test Doubles**, CQS provides a valuable framework to decide how to design and test interactions between the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)) and its collaborators. Different types of Test Doubles align naturally with either Commands or Queries:

- **Commands** are often associated with side effects and require verification that the expected actions were performed. For instance:
	- **Spies** and **Mocks** are commonly used to test Commands since they allow you to verify interactions, such as method calls or parameter values.
	- **Dummy** objects might be used to satisfy the structural requirements of commands but play no active role in testing behavior.
- **Queries** are focused on returning data and are generally tested by verifying the returned values rather than interactions. For instance:
	- **Stubs** and **Fakes** are well-suited for testing Queries because they can simulate data or behavior without requiring complex setups or external dependencies.

** **Fakes** could be used as both **Command** or **Query** depending on the implementation and use.

### Why CQS matters in testing with test doubles

By adhering to CQS, you can design tests that are more focused and aligned with the role of the [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test):

- **Clarity in purpose**: Clearly distinguishing between Commands and Queries helps decide whether the test should verify interactions (for Commands) or outputs (for Queries).
- **Reduced test complexity**: Properly separating Commands and Queries allows you to use the appropriate Test Doubles, avoiding unnecessary overlap or confusion between their roles.
- **Improved test reliability**: Following CQS ensures that tests are focused on one responsibility—either verifying side effects (for Commands) or validating data (for Queries)—leading to more robust and maintainable tests.

### Example of CQS with Test Doubles

Consider a system where a **NotificationService** is responsible for notifying users when an order is placed:

- **Command**: The `notify` method modifies the system's state by sending a notification. A **Mock** or **Spy** would be suitable here to verify that the notification is sent with the correct parameters.
- **Query**: The `getNotifications` method retrieves a list of sent notifications. A **Stub** or **Fake** would be more appropriate here, as they provide controlled data for testing without relying on the actual implementation.

### Test Smells and "The Mockery"

### What Is "The Mockery"?

**The Mockery** is a test smell that occurs when tests rely too heavily on mocks. This results in brittle, implementation-dependent tests that are hard to maintain. Instead of validating the actual behavior of the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)), these tests focus excessively on verifying interactions with collaborators. This problem often stems from developers defaulting to mocks for every dependency, usually because they aren't familiar with other test double options.

### Why Does "The Mockery" Occur?

This smell emerges when developers misuse or overuse mocks in their tests. Many dependencies would work better with alternatives like Stubs, Spies, or Fakes, but developers often reach for mocks without considering these options. Complex systems with many collaborators can pressure developers into writing interaction-heavy tests. When tests focus on implementation details rather than behavior, they become fragile and break during routine refactoring.

### Characteristics of "The Mockery"

- **Implementation Coupling**: Tests check internal mechanics instead of observable behaviors, breaking easily during refactoring.
- **Mock Proliferation**: Tests use too many mocks, making them complex and hard to follow.
- **Interaction Over Behavior**: Tests focus on method calls instead of meaningful outcomes.
- **Difficulty in Maintenance**: Tests become hard to understand and update due to tight coupling.
- **Knowledge Gap**: Developers use mocks by default because they don't know about other test doubles.

### Example of "The Mockery"

Consider the following example of excessive mocking in testing a `PaymentService`:

```java
public class PaymentService {
    private final HttpClient httpClient;
    private final Logger logger;

    public PaymentService(HttpClient httpClient, Logger logger) {
        this.httpClient = httpClient;
        this.logger = logger;
    }

    public String fetchReportData() {
        String data = httpClient.get("&lt;http://example.com/report&gt;");
        logger.log("Fetched data: " + data);
        return data;
    }
}
```

Example of a Test with "The Mockery" Smell:

```java
import org.junit.jupiter.api.Test;
import static org.mockito.Mockito.*;

class PaymentServiceTest {
    @Test
    void shouldFetchReportDataAndLogMessage() {
        // Arrange
        HttpClient mockHttpClient = mock(HttpClient.class);
        Logger mockLogger = mock(Logger.class);
        when(mockHttpClient.get("&lt;http://example.com/report")).thenReturn("Sample&gt; Data");
        PaymentService paymentService = new PaymentService(mockHttpClient, mockLogger);

        // Act
        String result = paymentService.fetchReportData();

        // Assert
        assertEquals("Sample Data", result);
        verify(mockHttpClient).get("&lt;http://example.com/report&gt;"); // Mockery smell: unnecessary interaction verification
        verify(mockLogger).log("Fetched data: Sample Data"); // Mockery smell: redundant logging verification
    }
}
```

This test relies too heavily on mocks, tightly coupling it to how `PaymentService` is implemented. Even when the core behavior stays the same, simple changes to method structure or logging would cause test failures.

### Avoiding "The Mockery"

To prevent this smell, write tests that verify behaviors instead of implementation details. Use mocks judiciously, only when you need to validate essential interactions with collaborators. For example:

```java
@Test
void shouldFetchReportData() {
    // Arrange
    HttpClient stubHttpClient = mock(HttpClient.class);
    when(stubHttpClient.get("&lt;http://example.com/report")).thenReturn("Sample&gt; Data");
    Logger dummyLogger = message -&gt; {}; // Simplify by using a Dummy
    PaymentService paymentService = new PaymentService(stubHttpClient, dummyLogger);

    // Act
    String result = paymentService.fetchReportData();

    // Assert
    assertEquals("Sample Data", result); // Focus on the outcome, not the interaction
}
```

### Conclusion

Test Doubles are a cornerstone of automated testing, providing essential tools to isolate the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)) and validate its behavior. They represent not just a technical convenience but a strategic approach to building reliable, maintainable, and meaningful tests. Their true value lies in understanding **when**, **how**, and **which type** to use while avoiding common pitfalls like over-reliance on mocks.

### Key Insights from the Article:

- Test Doubles simplify testing by replacing external dependencies with predictable alternatives. This allows tests to focus purely on the System Under Test's behavior rather than its complex interactions. As noted in *Growing Object-Oriented Software, Guided by Tests*, *"Test Doubles are critical for isolating the System Under Test ([SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test)) while maintaining the focus on verifying expected behaviors without interference from dependencies."*
- Each Test Double type: Dummy, Stub, Spy, Mock, and Fake—serves specific purposes. Misusing them through limited understanding or habit leads to brittle tests that impede refactoring and obscure intent. This aligns with the advice in *The Art of Unit Testing*: *"Use Test Doubles thoughtfully. Every Double you introduce adds complexity to your tests, so ensure it serves a clear purpose.".*
- Over-mocking, exemplified by **"The Mockery,"** occurs when tests focus on verifying interactions rather than validating behaviors. This highlights the importance of choosing the right Test Double for each scenario. As emphasized in *The Art of Unit Testing*, *"Excessive use of mocks leads to tests that are brittle and closely tied to the internal implementation, which stifles refactoring and increases maintenance costs."*
- Command Query Separation (CQS) helps guide Test Double selection:
	- **Commands** (actions that modify system state) pair well with Spies and Mocks to verify side effects.
	- **Queries** (operations that retrieve data without altering state) work best with Stubs or Fakes.
- Each Test Double brings its own trade-offs:
	- **Dummies** simplify structural requirements without adding test logic.
	- **Fakes** provide realistic behavior for integration-like tests but need maintenance.
	- **Mocks** enable precise interaction verification but risk implementation coupling.
- Before introducing a Test Double, consider these key questions:
	- **Who is the SUT?** Misidentifying it can derail test focus.
	- **What is the behavior our test double is covering?** Test doubles replace behaviors not structures.
	- **What behavior are we verifying?** Tests should validate system behavior, not just collaborator interactions.

### Applying the Lessons Learned

- Choose Test Doubles wisely:
	- Use Dummies for structurally required but behaviorally irrelevant dependencies.
	- Choose Stubs to control [SUT](https://emmanuelvalverderamos.substack.com/p/what-to-test-the-subject-under-test) inputs without verifying collaborator interactions.
	- Apply Spies to verify side effects while maintaining real implementations.
	- Use Mocks sparingly for precise behavior-driven development scenarios.
	- Employ Fakes for realistic, stateful dependency simulations in isolated environments.
- Match Test Doubles to their purpose:
	- Use Mocks or Spies for Commands to verify state changes and external interactions.
	- Apply Stubs or Fakes for Queries to validate return values independently.
- Recognize and avoid patterns like **"The Mockery"** to maintain robust, focused tests. Prioritize simplicity and clarity, emphasizing test validation over implementation mechanics.

### Final Thoughts

Test Doubles transcend their role as technical tools—they enable thoughtful design and behavior-focused testing. Understanding their capabilities helps developers create tests that both verify functionality and serve as living documentation. As noted in *Growing Object-Oriented Software, Guided by Tests*: *"Well-written tests act as living documentation for the behavior of the system, enabling better communication among team members and stakeholders."*

By carefully considering what to test, how to test it, and when to use Test Doubles, you'll create a test suite that's both reliable and maintainable while following clean development principles. As *Unit Testing Principles, Practices, and Patterns* suggest: *"The goal is to write tests that guide design, not dictate it. Focus on the contracts and behaviors rather than implementation details." *
